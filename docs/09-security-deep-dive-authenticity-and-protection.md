# 09 - Security Deep Dive: Authenticity and Protection

Security is not an afterlife thought in Gost; it's a core modular layer. This guide explores the advanced security features implemented to protect your data and your users.

---

## 1. Cryptographic Utilities (`common/security`)

Gost provides two main cryptographic utilities to handle data integrity and confidentiality.

### 1.1 Password Hashing (Bcrypt)
Location: `/src/common/security/hash.util.go`

Never store passwords in plain text. Gost uses **Bcrypt**, which is resistant to brute-force attacks due to its adaptive cost.
- `HashPassword(password string)`: Generates a secure hash.
- `ComparePasswords(password, hash string)`: Safely compares a plain password with a stored hash.

### 1.2 Field Encryption (AES-256-GCM)
Location: `/src/common/security/encryption.util.go`

For sensitive data that needs to be readable by the system (like encryption keys or PII), we use **AES-256-GCM**.
- `Encrypt(plainText, key)`: Returns a base64 encoded ciphertext.
- `Decrypt(cryptoText, key)`: Recovers the original string.
> [!IMPORTANT]
> Always set a strong `SECURITY_KEY` in your `.env` file for production!

## 2. Authentication Module (`modules/auth`)

The Auth module handles the identity lifecycle using **JWT (JSON Web Tokens)**.

### 2.1 JWT Strategy (Access & Refresh)
Location: `/src/modules/auth/auth.service.go`

Gost implements a dual-token strategy:
- **AccessToken**: Short-lived (e.g., 15m) for active communication.
- **RefreshToken**: Long-lived (e.g., 7d) to obtain new access tokens without re-logging.

### 2.2 Redis Blacklist (Logout)
When a user logs out, the token isn't just "forgotten" by the frontend. The `AuthService.Logout` method stores the token in **Redis** with a "blacklisted" flag. Subsequent requests with that token will be blocked by the Guard even if the JWT signature is valid.

## 3. Advanced Guards and RBAC

### 3.1 JwtGuard
Location: `/src/common/guards/jwt.guard.go`

This guard does more than just checking if a token exists. It:
1. Validates the Bearer format.
2. Checks the **Redis Blacklist**.
3. Verifies the JWT signature.
4. **Context Injection**: Maps the JWT claims (`sub`, `role`) into the Gin Context as `user_id` and `user_role`.

### 3.2 RolesGuard (RBAC)
Location: `/src/common/guards/roles.guard.go`

Implements **Role-Based Access Control**. You can protect routes based on the user's role:
```go
// Only admins can delete users
usersGroup.DELETE("/:id", guards.JwtGuard(), guards.RolesGuard("admin"), controller.Delete)
```

---

## 4. Rate Limiting Interceptor

Location: `/src/common/interceptors/rate-limit.interceptor.go`

To prevent DDoS and Brute Force, Gost includes a **Rate Limiter** powered by Redis. It tracks requests per IP and route.
If a user exceeds the limit, they receive a `429 Too Many Requests` error generated by the centralized `FormattedErrorGenerator`.

---

## 5. Standardized Error Responses

Location: `/src/common/utils/formatted-error-generator.util.go`

To maintain a consistent API contract (similar to NestJS's `HttpException`), use the `FormattedErrorGenerator`:
```go
utils.FormattedErrorGenerator(c, http.StatusUnauthorized, "Unauthorized", "Invalid credentials")
```
This ensures every error response follows the `{ "statusCode": X, "error": "...", "message": "..." }` pattern.

---

*Tip: Combine `JwtGuard` with `RateLimitInterceptor` on sensitive routes like `/login` and `/reset-password` for maximum protection.*
